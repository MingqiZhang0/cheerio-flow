# Cheerio Flow v0.1.6 Plan — Atomic Save + Storage Operation Console

**Status:** Plan  
**Date:** 2026-07-01  
**Baseline:** v0.1.5 — Group Folder Migration / Data Safety Foundation  
**Target version:** v0.1.6  

---

## 1. Purpose

v0.1.5 established seven layers of data safety defense: classification-based load routing, persistence gating, v1/v2 save routing with stale file quarantine, full backup with manifest, staging-based restore with rollback, explicit migration with dry-run and rollback, and UI-level safety guards (error type labels, stale report clearing, Ctrl scope).

However, the **largest known controllable risk** in v0.1.5 is the absence of an Atomic Save Layer. The current `write_json` function writes directly to the target path. If the process crashes, the system loses power, or the disk encounters an error mid-write, the target file may contain a partial, unreadable JSON payload. This is the same class of risk that WAL and atomic-rename patterns address in database systems.

v0.1.6 upgrades the save path from direct-write to **write temp → flush → fsync → atomic rename → verify**. Simultaneously, it introduces a **StorageEvent / Operation Log** as an observability layer — because an atomic save that you cannot see or audit provides no confidence to the user. A **Storage Operation Console**, opened from the status bar, gives the user a read-only view of recent storage events.

The principle is: **design the event model first, then implement Atomic Save within that observable framework.** If Atomic Save is built first and Operation Log is retrofitted later, the event emission points must be threaded through code that was not designed for them — that is rework.

**Key architectural constraints:**

- `AtomicWriteHelper` is a write-safety layer — it does not change *what* is written, only *how*.
- `StorageEvent` is an observability layer — it does not control save logic.
- `Storage Console` is a user-facing window — it does not execute dangerous operations.
- Console does not bypass existing Storage UI (Storage drawer remains the primary interaction point).
- Console does not replace Recovery Center (future v0.2.0).

## 5. AtomicWriteHelper Design

### 5.1 Pseudocode flow

```
fn atomic_write_json(target: &Path, data: &T) -> Result<(), String>
where T: Serialize
{
    // 1. Validate target path
    ensure_parent_directory_exists(target)?;
    ensure_not_symlink(target)?;

    // 2. Derive temp path in same directory
    let temp = target.with_file_name(
        format!(".{}.tmp", target.file_stem())
    );

    // 3. Serialize to temp file
    write_json(&temp, data)?;

    // 4. Flush and fsync temp file
    flush_and_fsync(&temp)?;

    // 5. Atomic rename (same volume → atomic on NTFS)
    fs::rename(&temp, target)?;

    // 6. Read-back verify
    let written: T = read_json(target)?;
    verify_invariants(&written)?;

    // 7. Emit SaveCommitted event
    emit_storage_event(StorageEvent::SaveCommitted { ... });

    Ok(())
}
```

### 5.2 Temp file naming convention

```
Target:  CheerioFlowData/projects/ungrouped/project-xxx.json
Temp:    CheerioFlowData/projects/ungrouped/.project-xxx.json.tmp

Target:  CheerioFlowData/app-state.json
Temp:    CheerioFlowData/.app-state.json.tmp
```

**Rules:**
- Temp file always in the **same directory** as target (required for atomic rename on NTFS).
- Temp file starts with `.` (hidden on Unix; on Windows, the leading dot has no special meaning but loader filters by extension).
- Temp file ends with `.tmp`.
- Loader must **ignore** files ending with `.tmp` (see Section 5.4).

### 5.3 Verify invariants after rename

After reading the target back, verify at minimum:

| Target type | Must verify |
|---|---|
| `app-state.json` | Valid JSON, `dataVersion` field present and is 1 or 2 |
| Project JSON | Valid JSON, `id` field present, `id` matches expected project ID |
| `groups.json` | Valid JSON, array of objects with `id` fields |
| Backup manifest | Valid JSON, `backupId` field present |

If verify fails → emit `SaveFailed` event. The old target (if it existed before this write) was already replaced by the rename — this is a **known limitation** of the current design. In practice, if `write_json` and `fsync` succeed and `rename` is atomic, the target should be valid. The read-back verify is a defense-in-depth catch for filesystem-level corruption, not for the common case.

### 5.4 Loader must ignore temp files

**This is the critical safety invariant for stale temp files.**

The v1 and v2 project loaders (`load_v1_project_files`, `load_v2_project_files`) already filter by `.json` extension. Files ending in `.tmp` will not match. **No code change required for the loaders** — but this invariant must be verified by test.

If a future version supports non-JSON project file extensions, the loader must explicitly skip `*.tmp` files.

### 5.5 Stale temp cleanup

Stale temp files (`.tmp` files older than 1 hour) should be cleaned up on startup. This is a best-effort hygiene operation, not a safety-critical one.

If a `.tmp` file is present and the target file also exists and is valid, the temp is stale and can be deleted. If the target is missing or invalid and a `.tmp` exists, the temp may represent a partially-saved file — the user should be notified, and the file should be **quarantined** (moved to `.cheerio/stale-temp-files/`), not deleted.

### 5.6 Important correction on acceptance criteria

> **Do NOT use "no .tmp file left after kill-process" as an acceptance criterion.**

The correct criteria are:

1. The old target file is not corrupted or truncated after an interrupted save.
2. Temp files are not loaded as real project files by the loader.
3. Temp files can be identified as stale by startup hygiene or future Recovery Center.
4. Stale temp files do not cause incorrect loading, incorrect saving, or data loss.
5. A stale temp file is recoverable evidence, not silent garbage.

## 6. Windows / Tauri / Rust Notes

### 6.1 Same-directory requirement

`fs::rename` is atomic on NTFS when source and destination are on the same volume. It is **not** atomic across volumes (the OS performs a copy + delete). Therefore, the temp file **must always be created in the same directory as the target file**.

### 6.2 NTFS rename-over-existing

On Windows, `MOVEFILE_REPLACE_EXISTING` (which Rust's `fs::rename` maps to) atomically replaces the target. This is a documented NTFS guarantee.

### 6.3 File locking by external processes

Antivirus software, cloud sync clients (OneDrive, Dropbox), search indexers, or backup tools may briefly open files and block rename. If rename fails:

- The temp file remains on disk.
- The old target file is untouched.
- The operation returns `Save failed`.
- No files are deleted.

### 6.4 Directory fsync

On POSIX, after renaming a file into a directory, you must `fsync` the directory to ensure the directory entry is durable. On Windows, `FlushFileBuffers` on a directory handle achieves a similar effect but requires the `windows-sys` crate or raw Windows API calls.

**v0.1.6 plan:** Directory fsync is a future hardening item. The first version relies on the atomicity of NTFS rename. If the system crashes between rename and the next directory metadata flush, the worst case is that the new file is lost but the old file was already replaced. However, since we write temp → fsync temp → rename, the temp file's data is already durable. After reboot, NTFS journal recovery should bring the renamed file into a consistent state. This is a calculated risk for v0.1.6.

### 6.5 Future hardening

| Item | When |
|---|---|
| `FlushFileBuffers` on parent directory after rename | Future version (v0.1.7 or v0.1.8) |
| `FILE_FLAG_WRITE_THROUGH` on temp file creation | Evaluate cost vs benefit |
| Explicit `FlushFileBuffers` via `windows-sys` | If implementation cost is low, include in v0.1.6; otherwise defer |

## 7. StorageEvent / Operation Log Design

### 7.1 Event structure

```rust
struct StorageEvent {
    id: String,            // UUID v4
    timestamp: String,     // ISO 8601
    severity: EventSeverity, // Info | Warning | Error
    operation: StorageOperation, // Load | Save | Backup | Restore | Migration | AtomicWrite | StorageRoot | Console
    phase: StoragePhase,   // Started | TempWritten | Fsynced | Renamed | Verified | Committed | Failed | Blocked | Rollback
    message: String,
    details: Option<String>,
    related_path: Option<String>,  // relative to data root, e.g. "projects/ungrouped/project-id.json"
    error_kind: Option<StorageErrorKind>,  // Load | Save | Restore | Migration
    workspace_id: Option<String>,
    data_version: Option<u32>,
}
```

### 7.2 Event severity

| Severity | Meaning |
|---|---|
| `Info` | Normal operation succeeded or is in progress |
| `Warning` | Operation completed but with caveats (e.g., stale temp cleaned) |
| `Error` | Operation failed (load failed, save failed, etc.) |

### 7.3 Event enumeration (minimum coverage)

```
LoadStarted
LoadCommitted
LoadFailed              ← error_kind = Load

SaveStarted
SaveTempWritten
SaveFsynced

SaveFailed              ← error_kind = Save

RestoreVerified
RestoreActivated
RestoreFailed           ← error_kind = Restore

```

### 7.4 Privacy note

v0.1.6 does **not** persist events to disk. The ring buffer lives in memory only and is cleared on app restart. This eliminates the privacy risk of file paths being permanently stored in operation logs.

When the user clicks **Copy Log**, they manually export the current buffer content. A future version may add an optional "redact paths" toggle in the Console UI, but this is out of scope for v0.1.6.

## 8. In-Memory Ring Buffer

| Property | Value |
|---|---|
| Storage | In-memory `VecDeque<StorageEvent>` |
| Capacity | 512 events |
| Overflow policy | Discard oldest event (FIFO eviction) |
| Persistence | None — cleared on app restart |
| Thread safety | Rust: `Mutex<VecDeque<StorageEvent>>` behind Tauri state |
| Access from frontend | Tauri command: `get_storage_events() -> Vec<StorageEvent>` |
| Clear | Tauri command: `clear_storage_events()` |

**Rationale for no persistence:**
- Reduces implementation complexity.
- Eliminates privacy risk from persisted paths.
- A future Recovery Center (v0.2.0) can design a privacy-conscious export with path redaction.

## 9. Storage Operation Console UI

### 9.1 Entry point

The existing status bar at the bottom of the app displays a summary derived from the latest `StorageEvent`:

```
Saved · 12:31:05
Saving...
Save failed · click for details
Load failed · click for details
Migration blocked · click for details
Backup created · 12:28:00
```

Clicking the status bar text opens the Storage Operation Console.

### 9.2 Console behavior

| Requirement | Detail |
|---|---|
| Open | Click on status bar save-status area |
| Close | Close button (X) or click outside |
| Display | Scrollable list of recent `StorageEvent` entries |
| Each entry shows | Timestamp, severity icon, operation, phase, message |
| Copy Log | Button copies all visible events as text to clipboard |
| Clear Console | Button clears in-memory buffer (confirmation not required — log is ephemeral) |
| Read-only | No Restore, Migrate, Delete, or Repair buttons inside Console |
| Auto-scroll | Console scrolls to latest event on open |
| Empty state | "No recent storage events." |

### 9.3 Console placement

The Console should be a **drawer or modal** that does not obscure the canvas. It should not be a persistent panel (to avoid competing with the Storage drawer for screen space). Suggested: slide-up panel from the status bar, or a centered modal.

### 9.4 Interaction with Ctrl radial menu

The Console must be added to the list of regions where Ctrl does **not** trigger the radial menu. This is achieved by ensuring the Console container has `[role='dialog']` or `dialog` — which is already covered by the expanded `isEditableTarget` guard from v0.1.5.

Additionally, the `isPointerOverCanvas` guard (from v0.1.5) naturally prevents Ctrl radial menu when the pointer is over the Console, since the Console is not inside `.canvas-wrap`.

## 10. Error Label Rules (Non-Regression)

v0.1.5 introduced `StorageErrorKind` to distinguish error types in the status bar. v0.1.6 **must not regress** these labels:

| Scenario | Status bar label | `error_kind` in event |
|---|---|---|
| Load failed (bad JSON, duplicate ID, etc.) | Load failed | `Load` |
| Save failed (disk full, permission, etc.) | Save failed | `Save` |
| Autosave blocked by persistence gate | Save blocked | `Save` (phase = Blocked) |
| Restore failed | Restore failed | `Restore` |
| Migration failed | Migration failed | `Migration` |

Storage Console events must use the **same** `error_kind` values. The event's `severity` should be `Error` for all of these, and the `message` should contain the user-facing error text.

## 15. Future Versions

v0.1.6 lays the observability and atomic-write foundation for the following roadmap:

```
v0.1.6  Atomic Save + Operation Log + Storage Console    ← THIS VERSION
v0.1.7  Snapshot + Manifest v2 + SHA-256 Content Checksum
v0.1.8  Single-Writer Workspace Lock
v0.2.0  Recovery Center + Data Safety Test Matrix
v0.2.1  Pseudo Server Root
v0.2.2  Push to Pseudo Server
v0.2.3  Pull + Conflict Detection
v0.2.4  GitHub Backup Adapter
v0.2.5+ Cloud-Assisted Adapter
```

v0.1.6 enables v0.1.7 (checksum) by ensuring that the files being checksummed are atomically written. It enables v0.1.8 (lock) by providing the atomic write primitive for the lock file itself. It enables v0.2.0 (Recovery Center) by providing the event log that the Recovery Center can query for diagnostic context.
