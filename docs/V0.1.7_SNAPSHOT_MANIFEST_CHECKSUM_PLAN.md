## Background

v0.1.6 completed **Atomic Save & Storage Operation Console**. Final read-only review passed.

v0.1.6 tag:


Current `main` includes post-v0.1.6 README / docs update commits:

v0.1.7 will build on v0.1.6's Atomic Save foundation to introduce an **active workspace snapshot manifest** with **SHA-256 checksums**, improving local data integrity observability without breaking existing load/save/backup/restore/migration semantics.

---

## v0.1.7 Overall Goal

v0.1.7 does **not** rewrite backup, restore, or migration. It does **not** implement a Recovery Center.

v0.1.7 targets:

1. Generate a snapshot manifest for the active workspace.
2. Compute SHA-256 checksums for active data files.
3. Write the manifest using atomic write.
4. Perform warning-mode verification on load.
5. Surface manifest / verify events in the Storage Operation Console.
6. Maintain v1 / v2 layout compatibility.
7. Ensure the manifest layer does **not** become a new data-availability risk.

## Core Design Decisions

### 1. Manifest File Location

```
CheerioFlowData/.cheerio/snapshot-manifest.json
```

Rationale:

- Explicitly identifies this as a **snapshot manifest**.
- Avoids confusion with `backup-manifest.json`.
- Resides in the `.cheerio` internal metadata directory.
- Does not pollute active project layout paths.

### 2. Manifest Coverage Scope

Only active workspace files are inventoried.

**Common (both layouts):**

- `app-state.json`
- `groups.json`

**v1 flat layout:**

- `projects/*.json`

**v2 group-folder layout:**

- `projects/ungrouped/*.json`
- `projects/groups/<group-id>/*.json`

Recursive collection of all JSON under `CheerioFlowData` is explicitly prohibited.

**Must be excluded:**

- `.tmp` files (atomic write temp files)
- `.cheerio/stale-project-files/` (quarantined stale files)
- Backups (`CheerioFlowBackups/`)
- `before-restore-*` directories
- `before-migration-*` directories
- `snapshot-manifest.json` itself
- `snapshot-manifest.json.tmp` (manifest atomic write temp)
- Non-active project JSON files
- Non-canonical project JSON files (stale paths not on the canonical path)
- Any unrelated metadata files

**Inventory generation timing:**

Manifest inventory collection MUST run **after** all active writes and stale quarantine have completed. The ordering for a typical save is:

1. Write canonical project JSON files.
2. Write `groups.json`.
3. Write `app-state.json`.
4. Execute stale quarantine (move non-canonical files to `.cheerio/stale-project-files/`).
5. Collect active inventory (only canonical paths remain).
6. Generate `snapshot-manifest.json`.

If inventory runs before stale quarantine, the manifest may include old non-canonical files that are about to be quarantined — producing false checksum mismatches on the next verification.

This is especially important for v2 group-move scenarios, where a project's path changes during the same save cycle: the old path must be quarantined before inventory runs.

### 3. Manifest Schema

Minimal schema. Versioned independently from `dataVersion`.

```json
{
  "manifestVersion": 1,
  "dataVersion": 2,
  "createdAt": "2026-07-01T12:34:56Z",
  "appVersion": "0.1.7",
  "generator": "Cheerio Flow v0.1.7",
  "layoutKind": "v2-group-folder",
  "projectCount": 1,
  "groupCount": 0,
  "files": [
    {
      "path": "app-state.json",
      "role": "app-state",
      "sha256": "e3b0c44298fc1c149afbf4c8996fb92427ae41e4649b934ca495991b7852b855",
      "sizeBytes": 208
    }
  ]
}
```

**Field notes:**

| Field | Meaning |
|-------|---------|
| `manifestVersion` | Snapshot manifest schema version. Starts at `1`. Independent of `dataVersion`. |
| `dataVersion` | Workspace data version: `1` (flat) or `2` (group-folder). |
| `createdAt` | ISO 8601 UTC timestamp of manifest generation. |
| `appVersion` | App semver string. |
| `generator` | Human-readable generator label. |
| `layoutKind` | `"v1-flat"` or `"v2-group-folder"`. Describes the active layout. |
| `projectCount` | Number of active project files inventoried. |
| `groupCount` | Number of groups in `groups.json`. |
| `files[].path` | **Relative** path from `CheerioFlowData/`. Forward-slash only (e.g. `projects/groups/grp-id/proj-id.json`). No absolute paths, no backslashes. |
| `files[].role` | `"app-state"`, `"groups"`, or `"project"`. |
| `files[].sha256` | Lowercase hex-encoded SHA-256. |
| `files[].sizeBytes` | File size in bytes. |

**Design constraints:**

- `manifestVersion` starts at `1` — this is the first snapshot-manifest schema.
- `dataVersion` remains `1` or `2` as set by the workspace loader.
- No `workspaceId` field — there is no reliable workspace ID. Do not derive one from absolute paths.
- All `path` values are **relative** from `CheerioFlowData/`. No user absolute paths. This avoids privacy exposure and cross-machine instability.
- All `path` values MUST use **forward-slash** (`/`) separators, never backslash (`\`). This ensures cross-platform consistency, deterministic sorting, and stable checksum reports. Windows backslash paths must be normalized before writing into the manifest.
- `files` array is sorted by `path` for deterministic output.

### 4. Checksum Approach

- Use Rust crate `sha2 = "0.10"`. Do not hand-roll SHA-256.
- Hex encoding can be implemented locally (no additional hex crate required).
- Checksum is computed over **raw file bytes** as they exist on disk.
- Checksum is computed **after** the active atomic save completes — it records what was actually written.
- JSON pretty-format differences affect checksum. This is acceptable: the checksum records the on-disk representation.

### 5. Manifest Write — Atomic

The manifest itself must use atomic write.

Reuse v0.1.6's `atomic_write_json`:

```
write temp  →  flush  →  sync_all  →  verify temp  →  rename  →  verify target
```

**Critical risk — manifest failure vs. active save success:**

If active JSON save succeeded but manifest generation or write fails, what should `save_database_to` report?

**This is a default design decision (not tentative):**

Active JSON save success MUST remain success. If active project JSON / `groups.json` / `app-state.json` have been saved successfully, but manifest generation / checksum / manifest write fails:

1. Do **not** roll back active data.
2. Do **not** disguise the active save as failed.
3. Do **not** display "Save failed" to the user, unless the active JSON save itself failed.
4. Report `manifest / failed` or an integrity warning.
5. If possible, return the warning through a `StorageReport.warnings` field or similar structured field.
6. If structured warning return is not feasible in v0.1.7, at minimum: log to stderr + emit a frontend warning event. Do not silently swallow the failure.
7. Slice 5 implementation MUST separately review this semantic before merging.

The v0.1.7 active save API contract:

```
Active data saved + manifest failed  →  Ok(save_report_with_warnings)
Active data saved + manifest ok      →  Ok(save_report)
Active data save failed              →  Err(save_error)
```

Manifest failure downgrades to a warning on an otherwise-successful save — it never upgrades a success into an error.

**Manifest-specific verify helper:**

The atomic write pipeline includes a "verify temp" step before rename. For manifest writes, do **not** rely solely on a generic `verify_any_json_value`. Add a dedicated helper:

```
verify_snapshot_manifest_json_value(value) -> Result<(), Vec<String>>
```

This helper must check at minimum:

| # | Check | Rationale |
|---|-------|-----------|
| 1 | `value` is an object | Reject arrays, strings, null |
| 2 | `manifestVersion == 1` | Only known schema version |
| 3 | `dataVersion` is `1` or `2` | Valid workspace data versions |
| 4 | `layoutKind` is `"v1-flat"` or `"v2-group-folder"` | Known layout kinds |
| 5 | `files` is an array | Structural integrity |
| 6 | Each file entry has `path`, `role`, `sha256`, `sizeBytes` | Required fields present |
| 7 | `path` is a relative path (not absolute) | Privacy and cross-platform safety |
| 8 | `path` does not contain backslash (`\`) | Cross-platform path normalization |
| 9 | `role` is `"app-state"`, `"groups"`, or `"project"` | Known role values |
| 10 | `sha256` is 64-char lowercase hex | Valid SHA-256 hex string |
| 11 | `sizeBytes` is a non-negative integer | Valid file size |

**Scope boundary:**

- These checks catch structural corruption in the manifest JSON before it replaces the existing manifest on disk.
- They do **not** perform deep semantic verification (e.g., checking that file count matches `projectCount`, or that every path actually exists on disk). Deep semantic checks run during load-phase verification (Slice 6), and even then only as warnings.
- A manifest that fails `verify_snapshot_manifest_json_value` must **not** be renamed into place — the old manifest is preserved.
- A corrupt manifest discovered during load verification is still only a **warning**, not a load blocker.

### 6. Verification Strategy — Conservative Warning-Mode

| Scenario | Behavior |
|----------|----------|
| Manifest exists and all checksums match | `verify / committed` — info event |
| Manifest **missing** | `verify / warning` — does **not** block load |
| Manifest **corrupt** (bad JSON) | `verify / warning` — does **not** block load |
| Checksum **mismatch** | `verify / warning` — does **not** block load |
| File in manifest **missing on disk** | `verify / warning` — does **not** block load |
| File on disk **missing from manifest** | `verify / warning` — does **not** block load |
| Bad JSON / duplicate ID / symlink blocker | Load **failed** — JSON gate still takes priority |

**Hard rules:**

- Checksum mismatch does **not** trigger automatic repair.
- Checksum mismatch does **not** auto-overwrite the manifest.
- Checksum mismatch does **not** auto-delete files.
- A load-failed workspace still has autosave blocked (v0.1.4 persistence gate preserved).

### 7. StorageEvent / Console Integration

Extend the existing `StorageEvent` type. Do **not** add a new panel or a Recovery Center.

**New operations:**

| Operation | Phase examples | Purpose |
|-----------|---------------|---------|
| `manifest` | `started`, `committed`, `failed` | Manifest generation and write |
| `verify` | `started`, `committed`, `warning`, `failed` | Manifest verification on load |

**New phase:**

| Phase | Meaning |
|-------|---------|
| `warning` | Non-fatal integrity observation (missing manifest, checksum mismatch, corrupt manifest) |

**New error kinds (optional — add only if needed for status bar labeling):**

| ErrorKind | Meaning |
|-----------|---------|
| `integrity` | Integrity-related warning or error |
| `manifest` | Manifest generation/write failure |
| `verify` | Verification failure or warning |

**Example event sequences:**

Manifest generation after save:

```
info  manifest / started    Manifest generation started.
info  manifest / committed  Manifest written.  related=.../snapshot-manifest.json
```

Or on failure:

```
error manifest / failed     Manifest write failed.  errorKind=manifest  details=...
```

Verification on load (all clear):

```
info  verify / started    Snapshot verification started.
info  verify / committed  All 3 file(s) verified.
```

Verification on load (mismatch):

```
info    verify / started    Snapshot verification started.
warning verify / warning    Checksum mismatch for projects/project-a.json.  errorKind=verify  details=expected abc... got def...
```

The Storage Operation Console displays these events as it does for all other operations. It remains read-only.

**Backend warning → frontend StorageEvent passing strategy:**

v0.1.6 `StorageEvent` is a **frontend in-memory buffer**. The Rust backend cannot directly call `appendStorageEvent`. For v0.1.7:

1. **Do not** introduce a Rust-side event stream in early slices.
2. **Do not** introduce Tauri event streaming solely for manifest / verify events — that is a cross-cutting infrastructure change and out of scope for v0.1.7.
3. **Preferred approach:**
   - Backend command results (save, load) return warnings / verification results as structured fields in their existing return types.
   - Frontend receives the command result and calls `appendStorageEvent` to feed the existing in-memory `StorageEvent` buffer.
   - This keeps the event buffer as the single source of truth and requires no new infrastructure.
4. If the current `StorageReport` / load result struct does not have a field to carry warnings, design a **minimal `warnings: Vec<String>`** (or equivalent) field in the relevant Slice — do not over-engineer it.
5. The Storage Console remains **read-only display**. It does not become a Recovery Center.
6. Manifest failure / verify warning events should use **`warning` severity** (not `error`), unless the manifest write command itself fails (which would be an `error`). The UI must not alarm users with red error badges for expected integrity observations.

## v1 / v2 Compatibility Requirements

1. v1 workspaces keep flat `projects/*.json` layout.
2. v1 autosave does **not** create `projects/ungrouped/`.
3. v1 autosave does **not** create `projects/groups/`.
4. v1 autosave does **not** auto-migrate to `dataVersion: 2`.
5. Writing `.cheerio/snapshot-manifest.json` into a v1 workspace is **metadata**, not a migration.
6. v2 workspaces keep group-folder layout.
7. v2 stale quarantine behavior is not regressed.
8. The manifest inventory helper must follow the same `classify_data_root` / loader logic that determines the active layout.

---

## Backup / Restore / Migration Boundary

v0.1.7 does **not** rewrite:

- Backup format
- Backup manifest (`backup-manifest.json`)
- Backup copy logic
- Restore staging
- Restore activation
- Migration staging (`build_v2_staging_data`)
- Migration apply (`apply_group_folder_migration`)
- Migration activation
- Stale quarantine

The v0.1.7 manifest covers only the **active workspace**.

If a backup copies `.cheerio/snapshot-manifest.json`, that is passive inclusion. Backup and restore correctness must **not** depend on the manifest.

Future versions may introduce a backup snapshot manifest. That is out of scope for v0.1.7.

### Slice 5 — Generate Manifest After Active Save Success

**Goal:**

- After active JSON save succeeds in `save_database_to_paths`, generate and write the snapshot manifest.
- Do **not** change v1 / v2 layout.
- Design and document the manifest-failure error/warning propagation semantics.

**⚠️ High-risk slice — requires separate review before merging.**

**Design must answer:**

1. If active JSON save succeeded but manifest write fails, what does `save_database_to` return?
2. Is the active save still considered successful?
3. How is the failure surfaced to the user?
4. Does the `StorageReport` struct need a `warnings` field?

**Default design decision (see also "Critical risk" in Core Design Decisions #5):**

- Active save success is preserved.
- Manifest failure is surfaced as a `manifest / failed` StorageEvent and a stderr log.
- If `StorageReport` can carry warnings without a breaking change, add a `warnings: Vec<String>` field.
- If not, stderr + StorageEvent is sufficient for v0.1.7.

**Recommended decomposition — Slice 5A and 5B:**

Slice 5 is the highest-risk slice in v0.1.7. To reduce risk, consider implementing it as two sequential sub-slices (or at minimum, two clearly separated commits within Slice 5):

**Slice 5A — Backend manifest generation after active save (no frontend changes):**

- After active JSON save succeeds, generate manifest in memory, compute checksums, write manifest via atomic write.
- Do **not** change frontend UI.
- Do **not** add new warning UI elements.
- The only invariant: active save semantics are not broken.
- If manifest write fails, stderr-log the failure and return `Ok(report)` for the active save.
- This sub-slice can be reviewed and merged independently — it adds manifest generation without any user-visible change.

**Slice 5B — Surface manifest write warnings to frontend:**

- Add a minimal `warnings` field to the command result (save / load return type) if needed.
- Frontend receives the command result and calls `appendStorageEvent` to emit `manifest / failed` or `verify / warning`.
- Storage Console displays the new events as part of the existing read-only event stream.
- Do **not** add a Recovery Center, repair buttons, or destructive actions.
- If `warnings` field cannot be added without a breaking change, use stderr log + a generic `appendStorageEvent` call from the frontend save handler as a fallback.

**⚠️ Reminder:** If keeping Slice 5 as a single slice, the implementer MUST NOT change backend save semantics and frontend event handling in one uncontrolled patch. These are two orthogonal concerns and must be reviewable independently.

**Tests:**

- Normal manifest generation after v2 save.
- Normal manifest generation after v1 save.
- Manifest write failure does not corrupt active data.
- Manifest write failure emits `manifest / failed` event.
- Manifest is not generated after a failed save.
- Manifest is not generated when `canPersist` is false.

### Slice 6 — Verify Manifest On Load, Warning Mode

**Goal:**

- On load, if `.cheerio/snapshot-manifest.json` exists, verify it.
- Missing manifest → `verify / warning`.
- Corrupt manifest → `verify / warning`.
- Checksum mismatch → `verify / warning`.
- File missing from disk → `verify / warning`.
- File on disk not in manifest → `verify / warning`.
- None of the above block the load.
- Bad JSON / duplicate ID / symlink blockers still cause Load failed.

**Load verification order:**

The following ordering is mandatory — the existing JSON load gate must run first and independently:

1. **Existing JSON load gate runs first** — bad JSON, duplicate ID, symlink detection, unsupported layout kind. Any of these can still produce Load failed.
2. **Only after active workspace load succeeds** does manifest verification run.
3. Manifest missing / corrupt / checksum mismatch / extra file / missing file → only produce `verify / warning`.
4. Verification warnings do **not** trigger autosave.
5. Verification warnings do **not** automatically regenerate the manifest.
6. Verification warnings do **not** automatically repair or delete files.

This ordering ensures the manifest layer never interferes with the existing load correctness gates.

**Tests:**

- Clean workspace: manifest exists, all checksums match → `verify / committed`.
- Old workspace without manifest → `verify / warning`, load proceeds.
- Corrupt manifest JSON → `verify / warning`, load proceeds.
- Checksum mismatch on one file → `verify / warning`, load proceeds.
- File in manifest but missing on disk → `verify / warning`, load proceeds.
- File on disk but not in manifest → `verify / warning`, load proceeds.
- Bad JSON in project file → Load failed (gate takes priority over manifest).
- Duplicate project ID → Load failed.
- Symlink in project path → Load failed.
- Load failed → autosave still blocked (`canPersist` gate preserved).

## Risk Register

| # | Risk | Mitigation |
|---|------|------------|
| 1 | Manifest treated as authoritative source, blocking valid old workspaces | Warning-mode only. Missing/corrupt manifest never blocks load. |
| 2 | Recursive inventory accidentally collects stale / backup / manifest self | Explicit exclusion list. Covered by Slice 1 tests. |
| 3 | Checksum mismatch escalated to Load failed | Conservative warning-mode design. Review during Slice 6. |
| 4 | Manifest write failure reported as active save failure | Slice 5 design must preserve active save success semantics. |
| 5 | Old workspace without manifest cannot load | Missing manifest is a warning, not an error. |
| 6 | v1 workspace misclassified as v2 due to manifest metadata | `layoutKind` is derived from `classify_data_root`, not self-reported. |
| 7 | Stale manifest after backup restore causes false warnings | After restore, old manifest either matches or triggers a warning. Both are acceptable. |
| 8 | Corrupt manifest blocks load or corrupts UI state | Corrupt JSON in manifest is caught and treated as a warning. |
| 9 | Storage Console warnings appear alarming to users | Use `warning` severity, not `error`. Message text should be clear and non-alarming. |
| 10 | Users misinterpret checksum as a zero-data-loss guarantee | README and docs must state the scope limitation. |
